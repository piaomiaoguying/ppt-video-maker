# TTS发言稿写作模板

> 本模板专为TTS文字转语音优化，面向短视频/中视频科普内容，注重口语化表达和朗读友好性

---

## 一、文档结构模板

### ⚠️ 严格的格式规范

**重要：Python脚本会严格按照以下规则解析发言稿，请务必遵守！**

#### 1. 页码标题格式（必须严格遵守）

```markdown
## 第N页：标题内容
```

**格式要求：**
- 必须是二级标题 `## `
- 必须以 `第` 开头
- 必须包含 `页：`（注意是中文冒号）
- 示例：`## 第1页：封面`、`## 第2页：今日议程`

**❌ 错误格式：**
```markdown
## 第一页：封面        # 错误：应该是数字
## 第1页:封面          # 错误：冒号应该是中文
### 第1页：封面        # 错误：应该是二级标题
第1页：封面            # 错误：缺少 ##
```

#### 2. 文本提取规则

脚本会提取每个页码标题之后、下一个页码标题之前的所有文本。

**脚本会自动处理：**
- 过滤代码块：`\`\`\`...\`\`\`` 之间的内容会被完全删除
- 移除加粗标记：`**文字**` → `文字`
- 移除括号内容：`【内容】` 会被删除
- 移除引用标记：`> 文字` → `文字`
- 移除列表标记：`- 文字` 或 `* 文字` → `文字`

**这意味着：**
- 列表项内容会被朗读，但列表标记会被移除
- 加粗的文字仍会朗读，只是没有加粗效果
- 代码块内容会被完全跳过（不会朗读）
- 引用的内容会被朗读

### 1. 文档头部

```markdown
# [视频主题] - 视频稿（TTS优化版）

> 面向[目标观众]的[视频类型]
> 预计时长：[时长范围]
> 发布平台：[平台名称]
```

### 2. 正文结构（按页面组织）

```markdown
---

## 第N页：[页面标题]

[开场白/过渡语]

[正文内容]

[结束语/过渡到下一页]
```

**完整示例：**
```markdown
## 第1页：封面

大家好！

今天我们来聊一聊"企业CLI化趋势"这个话题。

让我们开始吧。

---

## 第2页：今日议程

今天的分享分为三个部分：

第一部分，概念介绍。

第二部分，案例分析。

第三部分，总结展望。

好，让我们进入正题。
```

### ⚠️ 开场白建议(根据场景选择)

| 类型 | 示例 | 适合场景 |
|------|------|---------|
| 悬念钩子型 | `你知道吗？有一个正在悄悄改变行业的东西，叫做"XX"。` | 短视频/科普视频（黄金3秒抓眼球） |
| 痛点共鸣型 | `你是不是也遇到过这样的情况：XX？今天我来告诉你原因。` | 知识类/解决问题的视频 |
| 反常识型 | `很多人都以为XX是这样，但事实恰恰相反。` | 观点类/颠覆认知的视频 |
| 直接结论型 | `一句话总结：XX的核心就是三个字——简单、直接、稳定。` | 快节奏/信息密度高的视频 |
| 极简型 | `今天我们来聊"XX"。` | 任何场景/万能百搭 |

## 二、TTS友好写作原则

### ✅ 推荐做法

#### 1. **口语化开场与结尾**
- ✅ "你知道吗？XX正在成为行业的新趋势。"
- ✅ "今天我们来聊一聊XX这个话题。"
- ✅ "好了，以上就是今天的内容。"
- ✅ "如果你觉得这期内容有用，别忘了点赞收藏。"
- ✅ "下期我们来聊XX，记得关注。"

#### 2. **自然过渡**
- ✅ "好，我们进入正题"
- ✅ "了解了概念，我们来看案例"
- ✅ "接下来我们看下一个话题"
- ✅ "刚才提到了X，现在我们来详细看看"

#### 3. **设问引导思考**
- ✅ "看到这个标题，你可能会有个疑问..."
- ✅ "这意味着什么呢？"
- ✅ "为什么会这样呢？我来解释一下"
- ✅ "这个疑问很正常。因为..."

#### 4. **分段清晰**
- 每段专注一个观点
- 段落之间留空行（脚本会保留空行）
- 使用短句，避免长难句
- 一个句子表达一个完整意思

#### 5. **脚本处理后的实际朗读效果**

根据脚本的过滤规则，以下是实际会被朗读的内容：

| 原始Markdown | 脚本处理后 | 是否朗读 |
|-------------|-----------|---------|
| `**重点内容**` | `重点内容` | ✅ 朗读 |
| `> 引用内容` | `引用内容` | ✅ 朗读 |
| `- 列表项` | `列表项` | ✅ 朗读 |
| `* 列表项` | `列表项` | ✅ 朗读 |
| `【注释内容】` | （删除） | ❌ 不朗读 |
| ` ```代码块``` ` | （删除） | ❌ 不朗读 |
| `普通文本` | `普通文本` | ✅ 朗读 |

**实用技巧：**
- 不想朗读的内容放在 `【】` 中（如页码、备注）
- 代码示例用自然语言描述，不要放在代码块中
- 列表项可以正常使用，脚本会移除标记但保留内容

#### 6. **数字口语化（核心规则）**

**⚠️ 重要原则：发言稿中必须使用汉字数字，禁止使用阿拉伯数字！**

TTS引擎对阿拉伯数字的处理存在歧义，为保证朗读准确，所有数字都应转换为汉字。

##### 6.1 数字"1"的读法规则

| 场景 | 阿拉伯数字 | ❌ 避免 | ✅ 正确写法 | 朗读效果 |
|------|----------|--------|-----------|---------|
| 数量词 | 1个 | 一个(阿拉伯) | **一个** | 正常 |
| 序号/编号 | 101室 | 101室 | **幺零幺室** | 逐字读"幺" |
| 电话号码 | 138xxxx | 138xxxx | **幺三八xxxx** | 逐字读"幺" |
| 年份 | 2015年 | 2015年 | **二零一五年** | 逐字读 |
| 特殊编号 | 110 | 110 | **幺幺零** | 逐字读"幺" |

**使用场景示例：**
```markdown
❌ 避免：
项目在15年上线 → TTS可能读"十五年"
房间号是101 → TTS可能读"一百零一"
拨打110报警 → TTS可能读"一百一十"

✅ 推荐：
项目在二零一五年上线
房间号是幺零幺
拨打幺幺零报警
```

##### 6.2 数字"2"的读法规则

| 场景 | 阿拉伯数字 | ❌ 避免 | ✅ 正确写法 | 朗读效果 |
|------|----------|--------|-----------|---------|
| 量词前(个/本/张等) | 2个人 | 二个人 | **两个人** | 用"两" |
| 序数/等级 | 2年级 | 二年级 | **二年级** | 用"二" |
| 整十/整百 | 20/200 | 20/200 | **二十/两百** | 正常 |
| 千位 | 2000 | 2000 | **两千** | 用"两" |
| 编号逐字读 | 201室 | 201室 | **二零一室** | 逐字读 |

**使用场景示例：**
```markdown
❌ 避免：
我们有2个方案 → TTS可能读"二个人方案"(不自然)
会议在2楼举行 → TTS可能读"二楼"(可以，但建议统一)

✅ 推荐：
我们有两个方案
会议在二楼举行
房间是二零一室 → 编号逐字读
```

##### 6.3 年份读法

```markdown
❌ 避免：
15年、2026年、90年代

✅ 推荐：
一五年 或 二零一五年（根据上下文选择）
二零二六年
九十年代
```

**注意：**
- 如果上下文明确是年份，可以简化为"一五年"
- 首次提及时建议完整写法"二零一五年"
- 避免单独使用"15年"，容易读成"十五年"（数量词）

##### 6.4 特殊活动/品牌名称

```markdown
❌ 避免：
3.15晚会、985工程、211大学、315消费者权益日

✅ 推荐：
三幺五晚会（注意用"幺"不用"一"）
九八五工程
二幺幺大学
三幺五消费者权益日
```

**规律：** 品牌化、专有名词中的数字通常逐字读，且"1"读作"幺"

##### 6.5 大数字处理

```markdown
❌ 避免：
15000个用户、1000000次下载、1.5亿

✅ 推荐：
一万五千个用户
一百万次下载
一亿五千万 或 一点五亿
```

**规则：**
- 万以下：逐位转换（如 1234 → 一千二百三十四）
- 万以上：使用量级单位（如 15000 → 一万五千）
- 亿以上：使用量级单位（如 1.5亿 → 一亿五千万）

##### 6.6 小数和分数

```markdown
❌ 避免：
3.14、2.5倍、1/2、50%

✅ 推荐：
三点一四
两点五倍 或 两倍半
二分之一
百分之五十
```

##### 6.7 快速对照表

| 场景 | 错误示例 | 正确示例 |
|------|---------|---------|
| 年份 | 15年、2026年 | 一五年、二零二六年 |
| 特殊名词 | 3.15、985 | 三幺五、九八五 |
| 编号/序号 | 101、110 | 幺零幺、幺幺零 |
| 数量(量词) | 2个人、2本书 | 两个人、两本书 |
| 数量(序数) | 2年级、2等奖 | 二年级、二等奖 |
| 大数字 | 15000、1.5亿 | 一万五千、一亿五千万 |
| 小数 | 3.14、2.5 | 三点一四、两点五 |
| 百分比 | 50% | 百分之五十 |
| 分数 | 1/2、3/4 | 二分之一、四分之三 |
| 电话/编号 | 138xxxx | 幺三八xxxx |

#### 7. **列表逐项说明**
- ✅ "第一，封装。把复杂的..."
- ✅ "第二，标准化。统一的..."
- ✅ "我们一个一个来看：首先...其次...最后..."

### ❌ 需要规避的问题

#### 1. **网址朗读问题**

**问题：** TTS会逐字朗读网址，如"https://github.com/user/repo"会读成"H-T-T-P-冒号-斜杠-斜杠..."

**解决方案：**
- ❌ "访问 https://github.com/xxx 了解更多"
- ✅ "在GitHub上搜索'xxx项目'了解更多"
- ✅ "感兴趣的朋友可以在GitHub搜索相关项目"
- ✅ "详细内容请访问我们的项目主页"

**技巧：**
- 用平台名称代替网址（如"在GitHub上搜索"）
- 引导用户自行搜索关键词
- 如必须提及，用"项目主页"、"官网"等替代

---

#### 2. **代码片段朗读问题**

**重要：** 脚本会完全删除代码块内容（` ``` `之间的所有内容）！

**解决方案：**
- ❌ 直接粘贴代码块（会被删除，不会朗读）
- ✅ 用自然语言描述代码功能
- ✅ "一条简单的命令就能获取..."
- ✅ "只需要执行一个命令，指定标题和时间..."

**示例转换：**

❌ **避免使用代码块（会被删除）：**
```
执行以下命令创建笔记
（这里放代码块会被脚本完全删除，不会朗读）
```

✅ **推荐用自然语言描述：**
执行一条简单的命令，指定标题和标签，就能创建笔记。比如，你可以设置标题为"今日笔记"，添加"工作"和"重要"标签。

**如果必须展示代码：**
- 在PPT中展示代码截图
- 发言稿中用口语描述功能
- 或者把代码作为普通文本（不加代码块标记）

---

#### 3. **英文缩写朗读问题**

**问题：** CLI、MCP、API等缩写，TTS可能逐字母朗读或发音不准确

**解决方案：**

**方案一：首次出现时解释**
```markdown
✅ CLI，也就是命令行接口，是Command Line Interface的缩写。
✅ MCP，全称是Model Context Protocol，模型上下文协议。
```

**方案二：用中文替代（适合科普内容）**
```markdown
✅ "命令行接口"代替"CLI"（首次使用后可混用）
✅ "应用程序接口"代替"API"
```

**方案三：保留英文但注意上下文**
```markdown
✅ "JSON格式，也就是JavaScript对象表示法"
✅ "YAML格式，一种配置文件格式"

⚠️ 注意：技术类演讲中，部分缩写可保留，但需要：
- 首次出现时解释
- 上下文清晰
- 或用"这种格式"等指代
```

---

#### 4. **技术术语处理**

**问题：** 专业术语对普通听众不友好，且朗读生硬

**解决方案：**
```markdown
❌ 避免：
这个系统采用IPC机制进行进程间通信。

✅ 推荐：
这个系统采用IPC机制，也就是进程间通信技术。简单来说，就是让不同的程序能够互相"对话"。

✅ 更好：
这个系统采用了一种叫IPC的技术，让不同的程序能够互相通信。举个例子，你在终端执行命令创建笔记，主程序的界面会实时更新。
```

**技巧：**
- 先说术语，再解释含义
- 用"简单来说"、"举个例子"引导
- 类比日常生活场景

---

#### 5. **标点符号问题**

**问题：** 
- 省略号"..."可能被朗读
- 破折号"——"处理不一致
- 引号内容可能不自然

**解决方案：**
```markdown
❌ 避免：
这个工具很强大...

✅ 推荐：
这个工具很强大，但也有限制。

❌ 避免：
这就是——我们今天要讨论的主题

✅ 推荐：
这就是我们今天要讨论的主题。

❌ 避免：
这就是"双界面设计"的概念

✅ 推荐：
这就是"双界面设计"的概念。（TTS通常会处理引号）
或：这就是所谓的双界面设计概念。
```

---

#### 6. **特殊符号与格式**

**问题：** Markdown格式符号、特殊字符干扰朗读

**需要处理的符号：**
- 列表符号"-" "•"：写作时保留，TTS会忽略
- 强调符号"**"：写作时保留，但内容要口语化
- 分隔线"---"：写作时保留，TTS会跳过
- 括号"()"：内容会被朗读，注意括号内的句子要完整

**建议：**
```markdown
✅ 列表项直接写成完整句子：
- 第一，封装。把复杂的API调用封装成简单的命令。
- 第二，标准化。统一的输入输出格式。

✅ 强调内容也要口语化：
**核心观点：未来的软件需要同时服务两位主人。**

✅ 括号内补充说明：
2026年3月（也就是上个月），钉钉宣布全面CLI化。
```

## 三、结构化写作技巧

### 1. **页面/段落组织模式**

```markdown
## 第N页：[主题]

[开场钩子：用设问或场景引入]
→ "提到X，大家脑海中浮现的画面是什么？"

[核心观点：一句话概括]
→ "用一句话概括：..."

[详细展开：分点说明]
→ "第一，...第二，...第三，..."

[例子支撑：具体案例]
→ "我给大家看一个实际例子..."

[过渡引导：承上启下]
→ "了解了这些，我们来看下一个话题"
```

---

### 2. **内容层次设计**

#### 概念引入
```markdown
✅ 标准流程：
1. 设问引入 → "什么是CLI化？"
2. 一句话定义 → "用一句话概括：..."
3. 展开说明 → "具体来说，..."
4. 举例论证 → "举个例子，..."
5. 总结价值 → "这就是CLI化的价值"
```

#### 对比说明
```markdown
✅ 标准流程：
1. 提出对比 → "CLI和MCP有什么区别？"
2. 逐项对比 → "首先是本质区别...其次是调用方式..."
3. 总结选择 → "总结一下：它们各有适用场景"
```

#### 案例讲解
```markdown
✅ 标准流程：
1. 案例背景 → "飞书是字节跳动的企业协作平台..."
2. 具体做法 → "他们推出了Lark CLI，有几个特点："
3. 分点说明 → "第一，官方开源。第二，覆盖全面..."
4. 价值提炼 → "从这个案例可以看出..."
```

---

### 3. **节奏控制技巧**

#### 短句为主
```markdown
❌ 避免：
CLI化之后，AI只需要执行一条简单的命令，指定标题和时间，就能完成创建，这个过程简单、直接、稳定。

✅ 推荐：
CLI化之后，AI只需要执行一条简单的命令。指定标题和时间，就能完成创建。简单、直接、稳定。
```

#### 关键点重复
```markdown
✅ 重要观点重复强调：
"这就是CLI化在AI时代的价值——让AI能高效、稳定地操作软件。"

✅ 关键词重复：
"简单、直接、稳定。这三个词，就是CLI化的核心优势。"
```

#### 适当停顿（用分段实现）
```markdown
✅ 用分段标记自然停顿：

这意味着什么？

（段落分隔）

意味着未来的产品需要双界面设计。
```

---

#### 视频黄金3秒法则

**视频开头前3秒决定用户是否划走，必须直接抛出价值。**

```markdown
❌ 避免：
大家好，欢迎来到我的频道。今天我们来聊一个话题...
（前3秒没有任何信息，用户已经划走了）

✅ 推荐：
你知道吗？CLI化正在悄悄改变整个软件行业。
（第1秒就抛出悬念，抓住注意力）

✅ 推荐：
一句话总结：未来的软件必须同时具备图形界面和命令行界面。
（第1秒就给结论，信息密度高）
```

**技巧：**
- 不要寒暄、不要自我介绍、不要报目录
- 前3秒直接给：结论/悬念/痛点/反常识观点
- 开场白控制在10秒内，快速进入正题

Obsidian是一款非常流行的笔记软件。感兴趣的朋友可以在官网下载。

## 第2页：什么是CLI化？

用一句话概括：把软件的能力封装成AI可调用的标准命令。

举个例子：飞书有一个功能是"创建日历事件"。

如果没有CLI化，AI想要帮用户创建事件，需要打开浏览器、登录飞书网页版、找到日历模块、点击新建按钮、填写表单、点击保存。这个过程不仅复杂，而且容易出错。

CLI化之后，AI只需要执行一条简单的命令。指定标题和时间，就能完成创建。简单、直接、稳定。

### ⚠️ 脚本解析关键格式（必须遵守）

| 格式要求 | 正确示例 | 错误示例 |

**脚本处理规则：**
- ✅ `**文字**` → 朗读"文字"
- ✅ `> 引用` → 朗读"引用"
- ✅ `- 列表` → 朗读"列表"
- ❌ `【内容】` → 删除，不朗读
- ❌ `代码块` → 删除，不朗读

### 核心原则
1. **格式规范**：严格遵守 `## 第N页：标题` 格式
2. **口语化**：像聊天一样写作
3. **结构化**：清晰的段落和过渡
4. **简洁化**：短句为主，避免长难句
5. **人性化**：设问引导，案例支撑
